## 接到任务之后，先别急着干

上一篇我说过，Agent 就像一个特别靠谱的实习生——你给他一个目标，他不会闷头就冲，而是先搞清楚"我该怎么做"。

假设你跟这个实习生说："帮我规划一趟日本旅行。"

他会怎么做？直接订机票？当然不是。他会先列计划：第一天去哪、第二天去哪、预算多少、住哪里。**先想清楚，再动手。**

Agent 也一样。目标有了，任务书也写了，但 Agent 拿到这些之后到底怎么一步步去想、去做的？今天我们就来拆解：**Agent 到底是怎么"想"的？**

## Plan：先画地图，再走路

Plan 就是**把大目标拆成小步骤**的过程。

还是拿日本旅行举例。你的目标是"去日本玩一周"，Plan 就是那份旅行攻略：

- 第一天：到东京，逛浅草寺
- 第二天：去富士山
- 第三天：到大阪吃美食
- ……

有了攻略，你就不至于到了东京不知道该干嘛。

Plan 有两种：**静态的**和**动态的**。

静态 Plan 就是一开始把所有步骤都规划好，严格按计划走。动态 Plan 是边走边调——到了富士山发现下雨，那就改去温泉，不死磕。

**好的 Agent 系统通常是动态规划。** 因为现实世界充满不确定性，一条路走不通，得能灵活切换路线。死板的 Agent 走不远。

## Task：最小可执行单元

Plan 拆出来的每一步，就是一个 Task。

"去浅草寺"是一个 Task，"去富士山"也是一个 Task。但"玩遍日本"不是——**太粗了，没法执行**。反过来，"走到寺庙门口左转第三步"也太细了，决策负担太重。

**粒度很关键。**

做过前端的同学应该有感觉。Task 就像函数——**一个函数只做一件事，粒度适中**。太大的函数难维护，太小的函数到处调，代码反而更乱。Agent 的 Task 设计也是一样的道理。

## Spec：做事的规矩

说完 Plan，来说 Spec。

Spec 是对 **Agent 行为的约束和规则**。还是用旅行来类比——Spec 就是那份旅行守则：

- 预算不超过 5000 元
- 不坐红眼航班
- 每天最多 3 个景点
- 必须住离地铁站步行 10 分钟以内的酒店

**Plan 回答的是"做什么"，Spec 回答的是"怎么做时的规矩"。**

这两个概念特别容易搞混，我画个表格对比一下：

| 维度 | Plan（规划） | Spec（规格说明） |
|------|-------------|-----------------|
| 回答的问题 | 做什么？按什么顺序？ | 怎么做？有什么限制？ |
| 类比 | 旅行攻略 | 旅行守则 |
| 前端对应 | 组件渲染顺序/生命周期 | ESLint 规则 + TypeScript 类型 |
| 变化频率 | 随执行动态调整 | 相对稳定，像"宪法" |

**Plan 是动态的，走一步看一步随时改；Spec 是稳定的，不管你怎么走，规矩不能破。**

很多人写 Agent 提示词的时候，把 Plan 和 Spec 混在一起写，结果 Agent 行为混乱。记住：**Plan 是任务维度，Spec 是约束维度，分开写，Agent 才能既灵活又守规矩。**

## Chain of Thought：让 Agent 打草稿

Chain of Thought，简称 CoT，翻译过来就是"思维链"。

说白了就是：**让 Agent 先打草稿，再给答案。**

你小时候做数学题，老师是不是总说"写出解题过程"？不能直接写答案，得一步步推。这就是 CoT 的核心思想。

最经典的那句 Prompt 你一定听过：

> "Let's think step by step."

就这一句话，就能让大模型的推理准确率暴涨。因为它逼着模型**把中间过程暴露出来**，而不是直接蹦一个可能不靠谱的答案。

做前端的同学可以这么理解：**CoT 就像 debug 时一步步 console.log，把中间状态打印出来。** 你不打 log，bug 藏在哪你根本不知道。Agent 不"打草稿"，推理错了你也不知道错在哪一步。

## ReAct：想一步，做一步，看结果

ReAct 是我觉得 **Agent 最核心的运行方式**。

它的全称是 Reasoning + Acting——推理和行动交替进行。完整循环是这样的：

```
Thought（我想想该干嘛）
  ↓
Action（动手干）
  ↓
Observation（看看结果怎样）
  ↓
Thought（根据结果，想想下一步）
  ↓
Action → Observation → Thought → ...
```

**打个比方：做菜。**

你想想该放多少盐（Thought）→ 放盐（Action）→ 尝一口（Observation）→ 淡了，再加一点（下一轮 Thought）→ 加盐（Action）→ 再尝（Observation）→ 这次刚好，出锅！

**这就是 ReAct 循环：想 → 做 → 看 → 再想 → 再做 → 再看……直到任务完成。**

## ReAct 示例：帮我查天气订行程

光说概念太抽象，来看一个完整的 ReAct 循环。

假设你对 Agent 说：“帮我看看明天北京天气，如果下雨就帮我改签后天的高铁。”

Agent 的内心活动是这样的：

```
第1轮
Thought: 用户想知道明天北京天气，我需要调用天气工具查询
Action:  调用 get_weather(city="北京", date="明天")
Observation: 返回结果——“明天北京，小雨，气温12-18°C”

第2轮
Thought: 明天确认下雨了，用户说下雨就改签，我需要查后天的高铁
Action:  调用 search_train(from="北京", to="上海", date="后天")
Observation: 返回3趟高铁——G101(08:00)、G203(10:30)、G405(14:00)

第3轮
Thought: 有3趟车，用户没指定时间，选中间那趟比较合理，执行改签
Action:  调用 book_train(train_id="G203")
Observation: 改签成功，座位号12车15A

第4轮
Thought: 所有任务都完成了，给用户汇报结果
→ 输出最终回答：“明天北京有小雨，已帮你改签到后天G203次高铁，10:30发车，座位12车15A。”
```

**看到了吗？每一轮都是“想→做→看”三步。** Agent 不是一口气给出答案，而是一步步推理、调工具、看结果、再决定下一步。**中间任何一步结果不对，它都能在下一轮 Thought 里调整方向。** 这就是 ReAct 的威力——在行动中学习，在反馈中纠错。

## CoT 和 ReAct 的区别

这两个也容易搞混，记住一句话就行：

- **CoT 只"想"**——纯推理，不动手，打完草稿直接给答案
- **ReAct 是"想 + 做 + 看"**——推理和行动交替进行，中间能调工具、看结果、再调整

用公式表达：**ReAct = CoT + 工具调用 + 观察循环**

CoT 是 ReAct 的一部分，但 ReAct 比 CoT 多了"动手"和"看结果"这两个关键环节。

## 前端类比总结

给做前端的同学整理一份速查表，用你熟悉的东西对照理解：

- **Plan** 像组件的**生命周期钩子**——`componentDidMount` → `render` → `componentDidUpdate`，有序执行
- **Spec** 像 **ESLint 规则 + TypeScript 类型约束**——不管代码怎么写，都得遵守这些规矩
- **CoT** 像 **debug 时的断点调试**——一步步看中间状态，不让 bug 藏着
- **ReAct** 像**热更新开发循环**——改代码 → 看效果 → 不满意再改，循环往复直到满意

## 两个常见误区

**误区一：把 Plan 和 Spec 写在一起。**

很多人在写 Agent 提示词时，步骤和规矩混着写。结果 Agent 要么死板地执行计划不管约束，要么被约束绑住手脚不敢动。**Plan 是任务维度（做什么），Spec 是身份/约束维度（怎么做）。分开写，Agent 才能既灵活又守规矩。**

**误区二：把 ReAct 和 CoT 当成一回事。**

CoT 只是让模型"想清楚"，但想完之后该干嘛还干嘛，中间不会调工具、不会看结果。ReAct 是推理和行动交替，能真正跟外部世界交互。**一个是纸上谈兵，一个是实战演练。**

## 写在最后

今天把 Agent "思考"的几种方式拆了一遍：

- **Plan** 管"做什么"，把大目标拆成小步骤
- **Spec** 管"怎么做"，给 Agent 画红线定规矩
- **CoT** 让 Agent 想清楚再动手，不打无准备之仗
- **ReAct** 让 Agent 边想边做，在行动中学习调整

这些概念不是孤立的，一个好的 Agent 系统通常是 **Plan + Spec + ReAct** 的组合——有规划、有规矩、能行动、能调整。

学这些概念最大的感受是：**AI Agent 的设计思路，和软件工程中的很多思想是相通的。** 模块化、关注点分离、约束与自由度的平衡——这些我在前端工程化中反复实践的东西，在 Agent 领域换了个壳，但内核是一样的。

而且这些"思考方式"最终都要落地到一个循环里——ReAct 的"想→做→看"，说白了就是在一个 `while True` 里不断转圈。这个循环具体长什么样、怎么终止、怎么调工具，我会把自己手写 Agent 框架的实际代码放到下篇来聊。
