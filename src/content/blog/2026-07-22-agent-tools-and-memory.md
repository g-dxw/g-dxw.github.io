## 实习生光会想，还不够

上一篇我们聊了 Agent 是怎么"想"的——Plan、Spec、CoT、ReAct，全是脑子里的事。

但你想想，你雇了一个实习生，脑子转得快，计划写得漂亮，然后呢？**他坐在那儿不动。**

不行啊。他得拿起电话打给航空公司问航班，打开电脑查酒店价格，发封邮件帮你确认行程。**光会想不够，得会干活。**

Agent 也一样。前两篇讲了"大脑"，这篇讲讲 Agent 的"手"和"记忆"——它怎么调用工具、怎么在一个循环里不停地干活、怎么记住你说过的话。

## Agent Loop：那个停不下来的循环

Agent 干活不是一锤子买卖。它是一个**循环**：感知（拿到输入）→ 推理（想一想该干啥）→ 行动（调用工具）→ 观察（看结果）→ 循环……

![Agent Loop 流程：感知→推理→行动→观察循环](/images/agent-loop-flow.png)

这就是 **Agent Loop（代理循环）**——Agent 最核心的运行机制。上篇聊的 ReAct 循环——想→做→看→再想——落地到代码里，就是这个循环。

**前端同学秒懂：这不就是 `useEffect` 吗？** 你在 useEffect 里调接口、更新状态、状态变了又触发 useEffect——不加条件判断就是无限循环。Redux 也一样：**dispatch action → reducer 处理 → state 更新 → 重新渲染 → 用户操作 → 再 dispatch**，一圈一圈转，直到用户关了页面。

Agent Loop 同样的逻辑：收到指令 → LLM 推理 → 调工具执行 → 看结果 → 不够就继续，**直到满足终止条件**。

**必须有终止条件！** 写过 `useEffect` 的人都知道，在里面直接 `setState` 不加条件判断，组件会无限重渲染，浏览器直接卡死。Agent 也一样——**没有终止条件的 Agent 会无限循环，烧掉你的 token，干一堆没用的事。**

常见的终止条件：

- **目标达成**：机票订好了，任务完成
- **超出预算**：token 用完了 / 钱花完了
- **执行次数上限**：最多跑 10 轮，到了就停
- **失败退出**：连续出错 3 次，别挣扎了

我自己写 Agent 框架时加了一个**工具循环检测**：连续两轮调用完全相同的工具+参数，自动停止。大概率是卡住了，再跑也是浪费时间。

整个框架的核心，其实就是一个 `run()` 方法，内部调 `execute_loop()`。说白了就是一个 `while True`，每轮让 LLM 决定下一步干什么，调工具拿结果，再判断该继续还是该停：

```python
while True:
    response = llm.invoke(messages)      # LLM 推理，决定下一步
    tool_calls = response.tool_calls     # 看它要调什么工具
    if not tool_calls:                   # 没调工具 = 最终回答，停
        return response.content
    if len(history) >= max_iterations:   # 超过最大轮数，停
        return "达到最大迭代次数"
    # ...执行工具，把结果塞回 messages，继续下一轮
```

你看，就这么简单。**一个 while 循环，每轮做三件事：LLM 想、工具干、结果回来。** 终止条件写清楚，Agent 就不会跑飞。

那在这个循环里，Agent 到底用什么干活呢？

## Tool Use：Agent 的"工具箱"

**Tool Use（工具调用），也叫 Function Calling，是 Agent 从"光说不练"变成"能干活"的关键。** 没有工具的 Agent，就是一个特别能写的 Chatbot。

这个概念是理解 Agent 的关键，也是面试高频题。

打个比方。你家里有个工具箱：锤子、螺丝刀、尺子、扳手。装书架的时候，不是用手拧螺丝，而是**从工具箱里挑对工具，该用锤子用锤子，该用螺丝刀用螺丝刀**。Agent 的工具箱也差不多——搜索引擎查资料、API 接口调服务、代码执行器跑脚本、文件操作写日志。**前端同学可以这样理解：每个工具就是一个 API 调用。** `fetch()` 拉数据、`localStorage.setItem()` 存东西，Agent 做的事就是根据你的需求，自己决定调哪个 API、传什么参数。

**但工具是谁给的？是你定义的。** 工具不是 Agent 凭空发明的，是你给它准备好的——就像给新员工一份工具清单："这个是锤子，钉钉子用的；这个是螺丝刀，拧螺丝用的。"

怎么定义？**用 JSON Schema。** 比如一个查天气的工具：

```json
{
  "name": "get_weather",
  "description": "查询指定城市的天气信息",
  "parameters": {
    "type": "object",
    "properties": {
      "city": {
        "type": "string",
        "description": "城市名称，比如'北京'、'上海'"
      }
    },
    "required": ["city"]
  }
}
```

说白了就三样：**名字**告诉 Agent"这个工具叫什么"，**描述**告诉它"能干什么"，**参数**告诉它"调用的时候要传什么"。description 写得好不好，直接决定 Agent 会不会选对工具——你写"查询天气"，用户问"明天北京下雨吗"它就知道该用；你写"工具1"，它完全懵了。

**前端类比：这就是 TypeScript 的函数签名。** `function getWeather(city: string): Promise<WeatherResult>` ——参数名、参数类型、返回值，全是给调用者看的。Agent 选工具也是一样的道理。

Agent 怎么知道该选哪个？**不是随机试，是基于你的问题做语义匹配。** 你说"明天北京下雨吗"，Agent 想的是：用户想知道天气 → `get_weather` 描述里写了"天气" → 匹配！→ 提取出"北京"作为 city 参数 → 调用，拿结果。

**关键点：Agent 是在理解语义，不是做关键词匹配。** 你说"我想出门要不要带伞"，它也能理解这是在问天气。这就是 LLM 的能力——它理解意图，不只是匹配关键词。

但有个坑：**工具不是越多越好。** 对大多数常见场景来说，**3-5 个聚焦的工具效果往往比塞 20 个杂的更好**。你给 Agent 塞 50 个工具，它光选择就懵了——就像工具箱里堆了 50 个工具，找个螺丝刀都费劲。**少而精，比多而杂强太多。** 而且工具之间的职责边界要清晰，一个叫 `get_weather` 一个叫 `check_climate`，Agent 会纠结该用哪个。

## Memory：Agent 怎么记住你说过的话

Agent 干活时需要记住很多东西。记忆分两种。

**第一种：短期记忆，像便签本。** 就是当前对话的上下文窗口。你说了一句话，Agent 记住了；再问一句，它还记着前面说的。随手记几笔，空间有限，写满了就得撕掉旧的。**前端类比：组件的 state**——页面在就有，刷新就没了。短期记忆有个硬伤：上下文窗口有长度限制，聊了 50 轮，最早的对话可能已经被"挤出去"了。

**第二种：长期记忆，像日记本。** 持久化存储，Agent 把重要信息写进向量数据库或者外部存储，下次用的时候再检索出来。认真记录，永久保存。**前端类比：`localStorage` 或者数据库**——关了浏览器数据还在，下次打开还能读。

记忆管理的核心挑战是：**什么该记住，什么该忘掉。** 全记住？存储爆炸，检索变慢，噪声变多。全忘掉？每次对话都从零开始，体验极差。好的记忆系统会做**筛选**：重要的用户偏好、关键的历史决策——留下；临时的闲聊、一次性的查询——丢掉。跟前端的缓存策略一个道理：不是什么都缓存，只缓存高频访问、计算成本高的数据。

## Context：不是越多越好

**Context（上下文）是 Agent 当前能访问到的全部信息。** 它包括：

- 系统 Prompt（Agent 的"人设"和规则）
- 用户输入（你说了什么）
- 工具返回的结果（搜到了什么）
- 历史对话记录（之前聊了什么）

很多人觉得：上下文越多越好，把所有信息都塞进去，Agent 不就聪明了吗？

**错。核心原则是：信息密度 > 信息量。**

打个比方。你开会的时候，桌上放了 200 页报告，关键数据藏在第 137 页的一个表格里。你找得到吗？很难。**但如果桌上只有 5 页精炼摘要，关键信息一目了然，决策效率高得多。**

**前端类比：就像组件的 props。** 你传 30 个 props 进去，组件能维护吗？传 3 个关键的 props，组件干净利落。

实测下来的普遍经验是：**几千 token 的精炼上下文，效果往往优于堆几万 token 的信息堆砌。** 信噪比才是关键。

这也解释了为什么 Memory 和 Context 是一体两面的——**Memory 决定"存什么"，Context 决定"给 Agent 看什么"。** 存得再多，如果 Context 里塞的全是噪声，Agent 照样做不好决策。

## RAG：开卷考试

最后一个重要概念：**RAG（Retrieval-Augmented Generation，检索增强生成）。**

核心思想特别简单：**先查资料，再回答。**

类比：**开卷考试。** 你不需要把所有知识都背下来，但你得知道去哪本书、哪一页能找到答案。

工作流程就四步：

1. 用户提问："公司的报销流程是什么？"
2. 系统去检索相关文档（从公司内部知识库中找到报销制度文档）
3. 把检索到的文档塞进上下文
4. LLM 基于文档内容生成回答

**前端类比：这就是你做的搜索功能啊。** 用户在搜索框输入关键词 → 调 API 获取匹配的结果 → 展示在页面上。RAG 做的事本质上一样，只不过"展示"这一步变成了"让 LLM 读完后用自己的话回答"。

实际场景：让 Agent 回答公司内部文档的问题。Agent 不需要把所有文档都"背下来"（塞进模型参数里），只需要**会"查"**——知道去哪找、找到后怎么理解。

这也是为什么 RAG 现在几乎成了 Agent 产品的标配。它解决了一个根本问题：**模型的知识是静态的，但你的业务数据是动态的。** 公司制度改了、产品更新了、价格变了——你不可能每次都重新训练模型，但你可以让 RAG 去查最新的文档。

RAG 和 Memory 的关系也很紧密。长期记忆里的信息，往往就是通过 RAG 检索出来的。你之前聊过什么、Agent 之前做过什么决策——这些信息存在向量数据库里，下次需要的时候检索出来塞进上下文。**RAG 不只是用来查外部文档的，它也是 Agent 调取长期记忆的手段。**

## 两个常见误区

**误区1：工具越多 Agent 越聪明。**

错。前面说过了，生产环境 3-5 个工具最佳。太多工具降低选择准确率，Agent 反而更容易选错工具、走弯路。

**误区2：上下文越多越好。**

错。信号被噪声淹没，检索准确率和决策质量都会下降。精炼上下文是一种能力，不是一种偷懒。

## 写在最后

回顾一下今天的内容：

- **Agent Loop** 是运行机制——感知、推理、行动、观察，循环往复，必须有终止条件
- **Tool Use** 让 Agent 能干活——工具由开发者用 JSON Schema 定义，Agent 基于语义选择
- **Memory** 分短期和长期，管理好记忆是 Agent 好用的关键
- **Context** 讲究信息密度，不是越多越好
- **RAG** 让 Agent 会"查资料"，开卷考试总比闭卷强

说句实话，我刚开始啃这些概念的时候，心里是有点怵的。网上各种文章把 Agent 说得跟科幻片似的，什么"自主意识"、"通用智能"，看着就头大。但真正自己手写了一个 Agent 框架之后，突然就通了——**这些东西，前端全见过。** API 接口就是工具、TypeScript 函数签名就是工具描述、`state` 是短期记忆、`localStorage` 是长期记忆、搜索功能就是 RAG。再加上上篇聊的 Plan、Spec、CoT、ReAct——整个 Agent 的全貌就拼出来了。哪有什么黑科技，可能就是做了多年前端，这些模式早就见过了，只是换了层皮而已。
