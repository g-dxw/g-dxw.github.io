## 为什么做 LingWeave

常见的 AI 生图工具更像一张表单：写提示词、选择模型、点击生成，然后得到一张图片。这个流程适合单次尝试，但当我想继续沿用某个产品、人物或场景时，问题就出现了：上一轮用过什么参考图、改过哪些提示词、为什么会得到这个结果，很快就会散落在不同页面和聊天记录里。

我想要的不是再做一个生图表单，而是一个能保留创作上下文的工作区。

于是有了 **LingWeave（灵织）**：把提示词、参考素材、模型配置和生成结果放进同一张无限画布，用节点组织内容，用连线表达引用关系，让一张好结果可以自然地成为下一次生成的起点。

## 它现在能做什么

### 用画布组织完整的创作过程

画布支持文本、图片、视频、音频、生成配置和分组节点。可以自由缩放、连线、撤销重做，也可以管理多个画布项目并导入导出数据。

比起只保留最终图片，我更在意每一步是怎么产生的：它用了哪段提示词、引用了哪些素材、经过了什么配置，以及接下来又延伸出了哪些分支。

### 从一个结果继续生成

图片、文本、视频和音频都可以进入同一套节点工作流。生成结果不再是流程终点，而是可以继续连接到下一组提示词和配置中。

例如先确定产品主视觉，再引用产品图生成森林场景，接着同时引用产品和人物结果扩展城市画面。整个过程都留在画布上，方便对比、回退和复用。

### 沉淀提示词和素材

项目内置本地提示词库，也可以把常用提示词、参考图和生成结果保存到“我的素材”。这些内容默认保存在当前浏览器本地，支持导入导出，并可选用 WebDAV 同步。

### 让 Codex 协作操作画布

LingWeave 也提供了本地 Canvas Agent 和 Codex App 插件。连接之后，Codex 可以读取画布节点、整理关系、创建生成流程并触发生成。

这个能力是对手动画布的增强，而不是前置条件。即使不连接 Codex，所有核心创作功能也能独立使用。

## 本地优先，但不把接口锁死

LingWeave 以 Web 前端为主体，没有业务后端。画布、素材和生成记录默认保存在浏览器本地；AI 请求由浏览器直接访问用户配置的 OpenAI、Gemini 兼容接口，或部署环境提供的 Niffler 服务。

这意味着它适合个人使用和可信环境部署，也方便开发者替换自己的模型服务。不过项目目前仍在开发阶段，本地数据结构暂不保证向后兼容，不建议直接作为公网多人服务使用。

## 如何开始

最快的方式是打开[线上版本](https://g-dxw.github.io/lingweave/)，先从“快速上手”或内置案例了解节点之间的关系，再到“配置”中填写自己的 Base URL、API Key 和模型。

如果希望本地运行：

```bash
git clone https://github.com/g-dxw/lingweave.git
cd lingweave/web
npm install
npm run dev
```

项目也提供 Docker 部署方式，完整步骤可以查看[项目文档](https://g-dxw.github.io/lingweave/docs/overview/docker/)。

## 写在最后

LingWeave 最初基于 [basketikun/infinite-canvas](https://github.com/basketikun/infinite-canvas) 开发，在持续调整产品方向和架构之后由我独立维护，并继续遵循 AGPL-3.0 开源协议。

它现在还是一个很年轻的项目，但核心方向已经明确：**生成只是动作，连续推进才是工作。**

如果你也在用 AI 做连续场景、产品视觉、分镜或多模态内容，欢迎试用、提交 Issue，或者直接参与开发。
